1. Resumen ejecutivo

¿Qué es?

Sistema integrado para importar, validar, procesar y ejecutar nóminas de transferencias electrónicas masivas (TEF). Es un flujo de batch que funciona en 5 fases coordinadas.

Propósito

Permitir que bancos y empresas importen masivamente instrucciones de pago (nóminas) para distribuir dinero a múltiples beneficiarios de forma automática y controlada con trazabilidad total.

Novedad: ECM0003

Ejecución real de transferencias. Programa que procesa cada detalle individualmente, valida fondos, genera journals contables y mantiene trazas de cada TEF ejecutada.

Resultado

Transferencias automáticas, trazables, con control de errores por registro. Estado transaccional consistente desde importación hasta cierre.

3. Flujo Completo End-to-End

3.1 Ciclo de vida de una nómina (visión 360°)

ENTRADA (CSNOMINA - Archivo importado)
    ↓
[ITEFT050] IMPORTACIÓN Y VALIDACIÓN
    ├─ Valida tipo = 0018
    ├─ Valida estructura (cantidad, montos)
    ├─ Valida maestros (cliente, cuenta, bancos)
    ├─ Crea CINTNOH (cabecera) → Estado 04 o 03
    ├─ Crea CSMHNM (detalles)
    └─ Registra en NOHHST (histórico)
    ↓
[ITEFT060] AUTORIZACIÓN Y CAMBIO DE ESTADO
    ├─ Valida autorización del usuario
    ├─ Valida estado actual (04 → 06)
    ├─ Actualiza CINTNOH → Estado 06 (pendiente)
    └─ Registra en NOHHST
    ↓
[ECM0003] EJECUCIÓN DE TRANSFERENCIAS (LA MAGIA)
    ├─ Lee CINTNOH (cabecera) estado 08
    ├─ Lee CSMHNM (detalles) por cada transferencia
    ├─ POR CADA DETALLE:
    │  ├─ Valida cabecera
    │  ├─ Llama ETF6000 (motor de TEF)
    │  ├─ Recibe resultado (OK, Error, Warning)
    │  ├─ Actualiza CSMHNM (estado individual)
    │  ├─ Acumula contadores (OK, Error, Warning)
    │  └─ Registra en CSMTRACEIB (traza)
    ├─ Actualiza CINTNOH con contadores finales
    ├─ Estado final: 15 (OK) o 12 (con errores)
    └─ Registra en NOHHST
    ↓
[TEFT0070] PROCESAMIENTO Y CADUCIDAD
    ├─ Busca nóminas en estado 08-12
    ├─ Aplica reglas de caducidad
    ├─ Cierra o marca como abandonadas
    └─ Registra en NOHHST
    ↓
[PA_HISNOMI] AUDITORÍA Y CONSULTA
    ├─ Consulta histórico completo
    ├─ Revisa todos los cambios de estado
    └─ Proporciona trazabilidad 100%
    ↓
SALIDA (Nómina procesada con resultado)

Objetivo: Recibir archivo de nómina, validar estructura y datos maestros, crear registros de control e inicializar máquina de estados.

→
2. BuscaAbre archivo CSNOMINA
→
3. Lee cabeceraValida datos maestros
→
4. ConteoVerifica cantidad registros
→
5. GrabaCrea tablas de control

4.2 Validaciones de ITEFT050

# Validación Código Descripción Estado resultado
01 No duplicado 7448 Nómina no existe en CINTNOH 09 (Cancelada)
02 Archivo existe 2206 CSNOMINA accesible 01 (En creación)

4.3 Salida de ITEFT050

  • CINTNOH: Cabecera con estado 04 (OK) o 03 (parcial)
  • CSMHNM: Registros de detalle listos para procesar
  • CSMCTR: Registro de control con totales
  • NOHHST: Evento de importación registrado

5. ITEFT060: Autorización y Cambio de Estado

Objetivo: Validar autorización y cambiar estado de nómina para que pase a ejecución (ITEFT060 op. 0008 → ECM0003).

5.1 Flujo general

Operaciones en ITEFT060

000604 → 06 Envía a autorizar
→
000806 → 08 Ejecuta (llama ECM0003)
→
001208 → 12 Con errores
→
005708 → 57 Rechazada
→

5.2 Validaciones previas

  • Nómina existe: Búsqueda en CINTNOH
  • Estado válido: 04, 06, 08 según operación
  • RUT empresa: Coincide con solicitud
  • Autorización: Usuario + programa en CNTIN

5.3 Operación 0008 (CRÍTICA: Envío a ECM0003)

Cuando se ejecuta op. 0008:

  1. Prepara estructura ETF700001 con datos de nómina
  2. Actualiza CINTNOH → Estado 08 (en proceso)
  3. Llama SBMJOB a ECM0003 (en batch)
  4. ECM0003 procesa todas las transferencias
  5. Devuelve control con resultado

6. ECM0003: Ejecución Real de Transferencias TEF ⭐ (CENTRAL)

Objetivo: Procesar cada detalle de nómina, ejecutar TEF individual mediante ETF6000, acumular resultados y actualizar estado final. ESTE ES EL CORAZÓN DEL FLUJO.

6.1 Flujo general de ECM0003

Lógica de procesamiento de ECM0003

ENTRADA: CINTNOH (estado 08) + CSMHNM (todos los detalles)
    ↓
1. INICIALIZACIÓN
   ├─ Lee CINTNOH (cabecera)
   ├─ Valida que esté en estado 08
   ├─ Inicializa contadores:
   │  ├─ NOHNRA (transferencias aceptadas)
   │  ├─ NOHMRA (monto aceptado)
   │  ├─ NOHQRX (rechazo/warning)
   │  └─ NOHMRX (monto rechazo)
   └─ Detecta canal del usuario
    ↓
2. VALIDACIÓN DE CABECERA
   ├─ Cuenta débito existe (NOHACC)
   ├─ Cuenta activa (ACMAST = 'A')
   ├─ Moneda correcta (CLP)
   └─ Si falla → Estado 13, error crítico
    ↓
3. BUCLE POR CADA DETALLE (CSMHNM)
   ├─ Lee cada registro de detalle
   ├─ Prepara parámetros ETF6000:
   │  ├─ Cuenta débito (NOHACC)
   │  ├─ Banco destino (NODBCO)
   │  ├─ Cuenta destino (NODACC)
   │  ├─ Monto (NODMTO)
   │  ├─ RUT beneficiario (NODRAB)
   │  ├─ Nombre beneficiario (NODNAM)
   │  ├─ Concepto de pago (NODGCO)
   │  └─ Glosa compuesta (NOHMASCCT, etc.)
   │
   ├─ LLAMA ETF6000 (motor de transferencia)
   │  └─ Retorna: etfssa (status)
   │
   ├─ PROCESA RESULTADO:
   │  ├─ Si etfssa = ' ' (blank) → OK (NODSTS = 'A')
   │  │  └─ Incrementa NOHNRA, NOHMRA
   │  │
   │  ├─ Si etfssa = 'E' → Error (NODSTS = 'E')
   │  │  ├─ etfset = código error
   │  │  ├─ etfdes = descripción
   │  │  ├─ Incrementa NOHQRX, NOHMRX
   │  │  └─ NODDIB = descripción del error
   │  │
   │  ├─ Si etfssa = 'R' → Rechazo (NODSTS = 'R')
   │  │  ├─ Incrementa NOHQRX, NOHMRX
   │  │  └─ NODDIB = descripción rechazo
   │  │
   │  └─ Si etfssa = 'W' → Warning (NODSTS = 'W')
   │     ├─ Incrementa NOHQRX, NOHMRX
   │     ├─ NODIBS = 888
   │     └─ NODDIB = 'Validar con banco de destino'
   │
   ├─ Actualiza registro CSMHNM
   ├─ Registra traza en CSMTRACEIB
   └─ Continúa con siguiente detalle
    ↓
4. ACTUALIZACIÓN DE CABECERA
   ├─ CINTNOH.NOHNRA = total transferencias aceptadas
   ├─ CINTNOH.NOHMRA = monto total aceptado
   ├─ CINTNOH.NOHQRX = total rechazos/warnings
   ├─ CINTNOH.NOHMRX = monto total rechazado
   ├─ CINTNOH.NOHSTS = Estado final:
   │  ├─ '15' si no hay errores (NOHQRX = 0)
   │  └─ '12' si hay errores (NOHQRX > 0)
   └─ CINTNOH.NOHGLR = Descripción: "Procesada sin errores" o "con errores"
    ↓
5. REGISTRO DE AUDITORÍA
   ├─ NOHHST: Evento de ejecución
   ├─ ECM0004: Llamada para informe normativo I14
   ├─ RGAC001: Registro de transacciones en RGACAN (si aplica)
   └─ CSMTRACEIB: Log completo
    ↓
SALIDA: CINTNOH actualizada (estado 12 o 15)
        CSMHNM actualizada (cada detalle con resultado)
        NOHHST con evento de ejecución

6.3 Estados posibles en detalle (NODSTS)

Estado Significado Acción
A Aceptada / Exitosa Transferencia ejecutada correctamente
E Error en envío Fallo técnico (cuenta inválida, banco no disponible, etc.)
R Rechazo Banco rechazó la operación (fondos, límite, etc.)
W Warning Ejecutada pero con alerta (requiere validación con banco)
P Pendiente Sin procesar aún (inicial)

6.4 Contadores finales en CINTNOH

Campo Tipo Descripción

NOHMRA NUM(15,2) Monto total ACEPTADO
NOHQRX NUM(4) Cantidad rechazos / warnings
NOHMRX NUM(15,2) Monto total rechazado / warning
NOHSTS CHAR(2) Estado final: '15' (OK) o '12' (con errores)

6.5 Integración con ETF6000 (Motor de TEF)

ECM0003 NO ejecuta la TEF directamente. Llama a ETF6000 que es el verdadero motor de transferencia:

  • ECM0003: Orquesta, valida, acumula, gestiona estado
  • ETF6000: Ejecuta cada transferencia individual, genera journal contable

Relación ECM0003 ↔ ETF6000

ECM0003Lee CSMHNM detalle i
→
PreparaParámetros TEF
→
ETF6000Ejecuta real
→
RetornaStatus (A/E/R/W)
→
ECM0003Actualiza CSMHNM i

6.6 Consideraciones especiales en ECM0003

  • Nóminas H2H (0018): Usa NOHMASCCT como glosa origen, NODGCO como glosa destino
  • Canal: Se obtiene de CNTIN (tabla de autorización) para auditoría
  • Delay entre TEF: Si banco destino NO es '009', aplica sleep (usleep) para evitar congestión
  • RGAC001: Registra en tabla de contabilidad regulatoria para informe normativo
  • ECM0004: Llamada para actualizar informe I14 post-proceso

6.7 Flujo de error en ECM0003

Si hay error crítico en cabecera (validación inicial):

  • CINTNOH.NOHSTS = '13' (Error crítico)
  • Se registra en NOHHST
  • No se procesan detalles
  • Se retorna error a usuario

10. Validaciones Críticas

10.1 Niveles de validación

ITEFT050: Entrada

  • Archivo existe y accesible
  • Cabecera contiene datos
  • Tipo nómina = 0018
  • Usuario/canal autorizados

ITEFT050: Estructura

  • Cantidad detalles coincide
  • Monto total coincide
  • ≤ 2000 registros
  • Campos obligatorios

ITEFT050: Maestros

  • Cuenta débito existe
  • Cliente (RUT) existe
  • Beneficiarios válidos
  • Bancos conocidos

ECM0003: Cabecera

  • Nómina en estado 08
  • Cuenta débito activa
  • Moneda correcta (CLP)
  • RUT válido (mod 11)

11.3 Contadores acumulados en CINTNOH

Después de ECM0003 procesa todos los detalles:

ANTES de ECM0003:
├─ NOHSTS = '08' (en proceso)
├─ NOHNRA = 0
├─ NOHMRA = 0.00
├─ NOHQRX = 0
└─ NOHMRX = 0.00

DURANTE ECM0003 (por cada detalle):
├─ Si NODSTS = 'A' → NOHNRA++, NOHMRA += NODMTO
├─ Si NODSTS = 'E/R/W' → NOHQRX++, NOHMRX += NODMTO

DESPUÉS de ECM0003:
├─ NOHSTS = '15' (si NOHQRX = 0) o '12' (si NOHQRX > 0)
├─ NOHNRA = total aceptadas (ej: 45)
├─ NOHMRA = monto aceptado (ej: 1,234,567.89)
├─ NOHQRX = total rechazos (ej: 5)
└─ NOHMRX = monto rechazado (ej: 123,456.78)

12.1 Códigos de error principales

Código Descripción En programa Severidad
7448 Nómina duplicada ITEFT050 🔴 Crítico
2206 Archivo no encontrado ITEFT050 🔴 Crítico
2213 Tipo nómina ≠ 0018 ITEFT050 🔴 Crítico

12.2 Flujo de error en ECM0003

  • Error en cabecera (validación inicial): NOHSTS = '13' (error crítico), se registra, se retorna
  • Error en detalle: NODSTS = 'E/R/W', se acumula contador, se continúa con siguiente detalle
  • Resultado final: NOHSTS = '15' (OK) o '12' (con errores según NOHQRX > 0)

12.3 Recuperación

  • Error crítico de entrada: Retransmitir archivo corregido
  • Error de validación: Corregir datos maestros (clientes, cuentas)
  • Error de autorización: Revisar permisos en CNTIN
  • Error de detalle en ECM0003: Nómina queda en estado 12 (parcial), se puede reintentarla

13. Consideraciones Técnicas

13.1 Puntos clave de ECM0003

Es el corazón del flujo

ECM0003 es donde sucede la ejecución real de transferencias. Cada detalle se procesa individualmente y se acumula resultado.

No ejecuta directamente

Delega a ETF6000 (motor de TEF). ECM0003 orquesta, valida, acumula y gestiona estado.

Validación de cabecera

Si falla cabecera → NOHSTS = '13'. Si falla detalle → estado por detalle, continúa.

Contadores finales

Acumula aceptadas, rechazos, montos. Estado final depende de si hay rechazos (12) o todos OK (15).

13.2 Performance

  • Limitación 2000 registros: Evita saturación
  • Batch nocturno: TEFT0070 corre off-peak
  • Delay entre TEF: ECM0003 aplica sleep (usleep) si banco ≠ '009' para evitar congestión
  • Índices en CINTNOH, CSMHNM: (NOHTIN, NOHNFI) para búsquedas rápidas

13.3 Seguridad y trazabilidad

  • Autorización dual: Importación y ejecución separadas
  • Auditoría completa: NOHHST, CSMTRACEIB, NMINFOR (I14)
  • Integridad: Validaciones en múltiples niveles
  • Rechazo de duplicados: ITEFT050 verifica si nómina ya fue procesada

13.4 Integraciones externas

  • ETF6000: Motor de ejecución. ECM0003 lo llama por cada detalle
  • ECM0004: Actualiza informe normativo I14 post-proceso
  • RGAC001: Registra en RGACAN (contabilidad regulatoria)
  • CALTAXES: Calcula impuestos si aplica

13.5 Limitaciones conocidas

Sin transacción atómica

COMMIT = *NONE. Si falla intermedio, pueden quedar datos inconsistentes.

Concurrencia

Dos procesos simultáneos podrían validar sobre estado obsoleto.

Máximo 2000 registros

Nóminas grandes deben fragmentarse manualmente.

Sin reintentos automáticos

Errores de detalle requieren reintentos manuales.

13.6 Mejoras futuras sugeridas

  • Implementar transacciones atómicas (COMMIT = *CHG)
  • Locks optimistas en CINTNOH durante cambios de estado
  • Sistema automático de reintentos para errores
  • Particionamiento automático de nóminas > 2000
  • Dashboard en tiempo real de estados

Conclusión

El flujo de importación de nóminas es un sistema robusto, auditado y escalable. Su arquitectura modular con 5 programas especializados garantiza:

  1. ITEFT050: Importación validada
  2. ITEFT060: Autorización controlada
  3. ECM0003: Ejecución real, la pieza central
